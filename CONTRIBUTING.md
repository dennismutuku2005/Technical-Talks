# Contributing to Technical Talks 🤝

Thank you for your interest in contributing to our Technical Talks repository! This guide will help you contribute effectively to our developer community resource.

## 🎯 Ways to Contribute

### For Event Organizers
- Share event materials (slides, photos, feedback)
- Document event setup and lessons learned
- Add organizer guides and checklists
- Contribute templates for future events

### For Speakers & Presenters
- Upload presentation materials and slides
- Share code examples and demos
- Provide speaker notes and resources
- Add follow-up materials and links

### For Attendees
- Share workshop solutions and code samples
- Contribute questions and answers
- Add notes and summaries
- Provide feedback and testimonials

### For Community Members
- Improve documentation and guides
- Fix typos and formatting issues
- Suggest new features or improvements
- Help maintain and organize content

## 📋 Contribution Process

### 1. Before You Start
- Check existing content to avoid duplicates
- Review the repository structure and conventions
- Look at similar contributions for examples
- Read through relevant event documentation

### 2. Making Contributions

#### For Code and Technical Content
1. **Fork the repository** to your GitHub account
2. **Create a new branch** for your contribution
3. **Add your content** following our guidelines
4. **Test any code examples** to ensure they work
5. **Write clear commit messages** describing your changes
6. **Submit a pull request** with detailed description

#### For Documentation and Resources
1. **Navigate to the appropriate folder** (e.g., specific event directory)
2. **Add your files** using descriptive names
3. **Update relevant README files** if needed
4. **Follow our naming conventions**
5. **Submit your contribution**

### 3. Content Guidelines

#### File Naming Conventions
- Use descriptive names: `github-copilot-tutorial.md`
- Include dates when relevant: `workshop-2024-09-03.pdf`
- Use hyphens instead of spaces: `vscode-extension-demo.py`
- Be consistent with existing files

#### Content Standards
- **Clear and accurate:** Ensure information is correct and helpful
- **Well-organized:** Use proper headers, sections, and formatting
- **Community-appropriate:** Follow our code of conduct
- **Accessible:** Consider different skill levels and backgrounds

## 📁 Repository Structure Guide

```
├── events/                     # All events organized by year
│   └── YYYY/                  # Year folder
│       └── event-name/        # Specific event folder
│           ├── README.md      # Event details and information
│           └── resources/     # All event materials
│               ├── slides/    # Presentation materials
│               ├── code-samples/  # Code examples
│               ├── workshop-materials/  # Hands-on materials
│               ├── photos/    # Event images
│               ├── feedback/  # Community feedback
│               └── additional/  # Extra resources
├── templates/                 # Templates for future events
│   ├── event-template.md     # Standard event documentation
│   └── checklist-template.md # Organizer checklist template
├── CONTRIBUTING.md           # This file
└── README.md                # Main repository overview
```

## 🚫 What Not to Contribute

- Personal or sensitive information
- Copyrighted material without permission
- Spam or promotional content
- Offensive or inappropriate material
- Broken or untested code examples

## 📝 Pull Request Guidelines

### Writing Good Pull Requests
1. **Descriptive title:** Clearly explain what you're adding
2. **Detailed description:** Explain the purpose and value
3. **Reference issues:** Link to relevant issues or discussions
4. **Include screenshots:** For visual changes or new content
5. **Test your changes:** Ensure everything works correctly

### Example Pull Request Template
```markdown
## Description
Brief description of what this PR adds or changes.

## Type of Change
- [ ] New event documentation
- [ ] Resource addition (slides, code, etc.)
- [ ] Bug fix or improvement
- [ ] Template or tool

## Testing
- [ ] Verified all links work
- [ ] Tested code examples
- [ ] Checked formatting and spelling
- [ ] Reviewed for sensitive information

## Screenshots (if applicable)
Add screenshots to help explain your changes.
```

## 🎯 Quality Standards

### For Code Contributions
- Code should be clean, commented, and tested
- Include brief explanations for complex logic
- Use consistent formatting and style
- Provide usage examples when helpful

### For Documentation
- Use clear, concise language
- Include proper headings and structure
- Check spelling and grammar
- Add relevant links and references

## 🤝 Code of Conduct

- **Be respectful:** Treat all community members with kindness
- **Be collaborative:** Help others and accept feedback gracefully
- **Be inclusive:** Welcome contributors of all backgrounds and skill levels
- **Be professional:** Maintain high standards in all interactions

## 💡 Getting Help

If you need help with contributing:

2. **Open an issue** to discuss your contribution idea
3. **Ask questions** in discussions or issues
4. **Review existing pull requests** for examples
